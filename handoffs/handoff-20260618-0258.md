## 3. Known Issues (遗留问题)

### 🔴 核心未解决问题：iOS 本地循环导致跳动+loading
- **症状**：打开 codex 长 session（如 019ec076，3.3MB / 10 条消息含大量 tool output），消息列表频繁上下跳动 + loading 转圈反复出现
- **关键证据**：Mac 端 0 请求时仍跳动 → **iOS 本地循环**，非网络驱动
- **可疑但未证实**：有时 Mac 收到 14 次 get_session_messages / 60s（~2-4s 间隔，probe polling 特征），有时 0 请求 → 触发源不稳定，可能受 app 生命周期/重连/缓存状态影响
- **待验证假设**：删除的 probe 分支可能不是唯一触发源；`loadMessages` 可能在缓存命中路径（不发网络）反复执行，触发 `isLoadingMessages` 翻转 + messages 数组重建 → 渲染抖动
- **日志工具已就位**：`oscDiag.log`（app 沙箱 Application Support/）含调用栈，**未拉回**——next agent 第一动作

### ⚠️ 去 page 分页的副作用（已知接受）
- relay VPS `/etc/cccode-relay/relay.env` 从 2MB 改成 32MB（覆盖代码默认值）。如果将来重新部署且 env 文件被重置，会回到 2MB → 1009 复现。**部署时必须同步检查 env 文件**。
- 写 deadline 120s 削弱了写方向的半开检测（半开连接判死从 10s 拉到 120s）。半开检测现在主要靠 go-bridge 独立应用层心跳(10s) + relay 读 deadline(90s)。可接受。
- 超大 session（>25MB 业务 payload，加密+base64 后 >32MB）仍会 1009。当前 session 都 <3.5MB，安全。
- `servePaginatedMessages` + `transcriptindex` 整套分页代码保留未删（capability 关闭后走不到，但是死代码）。用户意图是「减少复杂性」，彻底删分页代码是后续可选工作。

### ⚠️ syslog 抓取工具失效
- `libimobiledevice`（`idevicesyslog`/`idevice_id`）在当前环境**无法检测设备**（`idevice_id -l` 返回空，`usbmuxd` not found）。CoreDevice 通道（`xcrun devicectl`）仍可用，但没有流式 syslog 子命令。
- **iOS 明文日志必须用 oscDiag 文件方案**（已实现）+ `devicectl device copy` 拉回，不能用 syslog。
