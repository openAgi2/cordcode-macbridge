## 结论

**暂不批准进入 P0 实现。**

修订版已经正确落实了辅助栏前移至 P0、按 `BridgeRuntimeStatus` 定义主 CTA、连接唯一目的地、自定义 Relay endpoint 可见性，以及 30/200 行的日志浏览模型。方向不需要重做。

但仍有两处与当前代码和算术事实直接冲突的规格：

1. 1180pt 被定义为 NSWindow 总宽度，却同时要求 1164pt 的内容宽度预算；该双列分支在 1180pt 会必然溢出或压缩主列。
2. 文档要求“复制支持信息”使用 `rawLogs`，这会推翻现有的脱敏支持信息契约，把原始日志意外混入本应安全的支持摘要。

修正下列阻塞项后即可批准 P0；日志复制项只阻塞 P2，但应现在改正文，避免实现者带着错误的安全要求进入后续阶段。

## 阻塞项

### B1：1180pt 阈值与内容宽度预算不成立（阻塞 P0）

修订稿 §3.1 定义：`wideSecondaryThreshold = 1180pt` 测量 NSWindow 总宽度；同时给出：

```text
availableContentWidth = windowWidth - sidebarWidth - dividerWidth - horizontalPadding × 2
```

按当前实现的实数代入：

```text
1180 - 180 (Sidebar) - 1 (Divider) - 60 (PageContainer 30×2) = 939pt
```

而同一节要求的最小双列预算为：

```text
900 (main) + 24 (gap) + 240 (inspector) = 1164pt
```

即在阈值点至少缺 **225pt**。此外 `PageContainer` 当前 `maxContentWidth` 仍为 880pt；即使 `GeometryReader` 判断为宽屏，若不显式放宽该容器，双列也无法得到 1164pt。

**必须采用以下二选一方案之一：**

1. **推荐：按 Workspace 内容区测量。** 将常量改为 `wideWorkspaceContentThreshold = 1164`（或加合理余量），在 `WorkspaceView` 内取得 Sidebar 之后、扣除 `PageContainer` 水平 padding 后的实际宽度；仅当 `availableContentWidth >= main + gap + inspector` 时渲染双列。`PageContainer` 必须允许 Workspace 在宽屏使用相同或更大的 `maxContentWidth`。
2. **若坚持以 NSWindow 总宽度测量：** 阈值至少为 `1164 + 180 + 1 + 60 = 1405pt`，并仍须将 Workspace 的 `PageContainer.maxContentWidth` 放宽到可容纳双列的尺寸。

traffic lights 不应纳入水平预算；它们不占用内容区横向宽度。低于最终阈值时必须直接单列，不能压缩 900pt 主列或 240pt 辅助栏。

### B2：支持信息不得包含 rawLogs（修正 P2 安全契约）

修订稿 §4.3 写“复制原始日志 / 支持信息复制必须使用最近 200 行完整 `rawLogs`”。这与当前 `DiagnosticsViewModel.copySupportInfo` 的明确职责冲突：它只构造 Bridge/连接/AI 工具的**脱敏摘要**，并刻意不包含 route id、token、密码或 endpoint 凭据；`copyRawLogs()` 才复制日志。

**必须改为：**

- `rawLogs`：读取的最近 200 行；“展开 200 行”和“复制原始日志”都使用它。
- `visibleLogs`：仅为界面渲染的 `rawLogs.suffix(30)` 或完整 `rawLogs`。
- `copySupportInfo`：继续生成脱敏摘要，**不包含 rawLogs**。

这既满足“原始日志可完整、原样获取”，也不把可能含运行细节的日志混入用户预期为安全摘要的复制动作。

## 非阻塞但应随 P0 / P2 修正的文字与实现约束

1. §0 仍写“连接状态是唯一动作入口”，§2 已正确改为“唯一目的地”。应统一为“唯一连接相关目的地”，避免与允许的 Toolbar / Relay 行两个按钮自相矛盾。
2. 文档引用 `PageContainer.horizontalPadding`，但当前实现将 `30` 写为私有字面量。P0 应先把它抽成公开的 `LayoutConstants.pageHorizontalPadding`，再让布局计算引用这个唯一 token，避免公式和实际 View 漂移。
3. “此 Mac 名称”不能只写“`bridgeDisplayName`（或现有配置源）”。当前 `WorkspaceView` 尚未持有名称；实现规格应明确传入 `SettingsViewModel.displayName` 或复用同一持久化 source，并定义空值时显示什么。不得新增网络探测。
4. 当前日志 UI 会经过 `DiagnosticsViewModel.displayLogLine` 截断超长行以便显示。P2 的“原样”应明确保障读取/复制为原文；如果要求屏幕也逐字符展示，需另行写明对横向滚动与性能的取舍，不能把两种语义混在一起。

- [ ] 文档和 `LayoutConstants` 采用 B1 的任一自洽方案，并为 Workspace 放宽 `PageContainer.maxContentWidth`。

- [ ] 主 CTA 状态表保持以真实 `viewModel.status` 为真值。

满足 B2 后即可按文档继续 **P2**。所有原有验证边界维持：定向 Swift 测试、macOS Release build、覆盖安装和 runtime 核对；UI automation、snapshot、simulator 或真机 UI 验收仍需 owner 明确授权。

## 最终建议

本次不要求改设计方向，只要求先修正宽度算术和支持信息安全边界。完成后，方案将具备可直接进入实施的尺寸契约、状态契约和诊断数据契约。
