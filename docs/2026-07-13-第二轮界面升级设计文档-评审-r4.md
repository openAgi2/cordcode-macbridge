## 剩余阻塞：宽屏尺寸契约尚不确定

### B1-r4：测量位置不能使用“或”

§3.1 的代码说明为：

```swift
let availableContentWidth = geometry.size.width // 在 PageContainer 内容区内测量，或扣除已知 padding 后
```

“在内容区内测量”与“在外层测量后扣 padding”是两条不同路径。`PageContainer` 当前把 30pt 水平 padding 写在内容的外层；若 `GeometryReader` 放在不同层，`geometry.size.width` 的含义不同。不能在实施规范中用“或”。

**必须改成唯一实现方式：**

```swift
// GeometryReader 放在 Workspace 的 PageContainer 内容内；该宽度已经是可用内容宽度。
let availableContentWidth = geometry.size.width
let isWide = availableContentWidth >= LayoutConstants.wideWorkspaceContentThreshold
```

或者选择外层测量，但必须写成唯一公式：

```swift
let availableContentWidth = geometry.size.width - 2 * LayoutConstants.pageHorizontalPadding
```

二者只能保留一种；`PageContainer` 也必须实际改用 `LayoutConstants.pageHorizontalPadding`，不能继续保留 `30` 字面量。

### B1-r5：1164pt 阈值与“推荐列宽”不一致

文档给出最小预算：

```text
900 + 24 + 240 = 1164pt
```

同时又推荐主列 920–940pt、辅助栏 260pt。后者的真实预算是：

```text
920 + 24 + 260 = 1204pt
940 + 24 + 260 = 1224pt
```

因此在 `wideWorkspaceContentThreshold = 1164` 的临界点，若实现者采用文档“推荐”尺寸，布局仍会短 40–60pt。

**必须选定一个可验证的策略：**

1. **推荐：固定最小值 + 有余量才增长。** `wideWorkspaceContentThreshold = 1164` 时，双列固定使用 main=900、gap=24、inspector=240；只有 `availableContentWidth >= 1204` 才把列提升到 920/260，随后按明确优先级继续增长。这样阈值公式和布局永远一致。
2. **更简单：采用推荐尺寸作为最小值。** 将阈值改为 1204（或 1224），并在该阈值下使用固定的 920/260（或 940/260）。

无论选择哪种，`PageContainer(maxContentWidth:)` 在 Workspace 宽屏模式必须至少接收同一个双列总宽度；否则即使判断正确，现有 880pt 上限仍会裁掉布局。
