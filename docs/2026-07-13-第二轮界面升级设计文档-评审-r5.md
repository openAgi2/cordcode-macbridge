# 第二轮界面升级设计文档 — r4 尺寸实现复核

日期：2026-07-13  
评审对象：`docs/2026-07-13-第二轮界面升级设计文档.md`（r4 尺寸契约收口版）  
前置评审：`docs/2026-07-13-第二轮界面升级设计文档-评审-r4.md`

## 结论

**暂不批准进入 P0；剩余一个 SwiftUI 容器层级的算术错误。**

r4 已把测量坐标、最小/舒适双列尺寸、日志安全边界和状态规则写得足够清楚。唯一未闭合点是：`PageContainer.maxContentWidth` 包含其两侧 horizontal padding，而 r4 又规定 `GeometryReader` 位于 padding 内。文档把 `maxContentWidth` 传为“主列 + gap + 辅助栏”会少 60pt，导致宽屏测量值不足阈值，双列分支不能在预期临界点触发。

这是实现细节的最后一处不自洽，不改变任何产品方向。

## 证据与计算

当前 `PageContainer` 的层级是：

```swift
content
    .frame(maxWidth: .infinity)
    .padding(.horizontal, LayoutConstants.pageHorizontalPadding) // 30 × 2
    .frame(maxWidth: maxContentWidth)
```

r4 同时规定 GeometryReader 放在 `content` 内，因此在 `maxContentWidth` 已命中上限时：

```text
geometry.size.width = maxContentWidth - 2 × pageHorizontalPadding
```

若照 r4 §3.1 / §3.1 的“匹配双列总宽”传入：

```text
maxContentWidth = 900 + 24 + 240 = 1164
geometry.size.width = 1164 - 60 = 1104
```

但 `wideWorkspaceContentThreshold = 1164`。结果是 `isWide` 永远为 false；即使窗口更宽，容器也先封顶在 1164，内部 GeometryReader 不可能达到 1164。

## 必须采用的最终修正

将“内层实际内容宽度”与“PageContainer 外层宽度”定义为两个不同 token：

```swift
// 内层 GeometryReader 可用内容宽度
static let workspaceWideContentThreshold: CGFloat = 1164 // 900 + 24 + 240
static let workspacePreferredContentWidth: CGFloat = 1204 // 920 + 24 + 260

// PageContainer 的 maxContentWidth，包含两侧 pageHorizontalPadding
static let workspaceWideContainerWidth =
    workspaceWideContentThreshold + 2 * pageHorizontalPadding // 1224
static let workspaceMaxContainerWidth =
    workspacePreferredContentWidth + 2 * pageHorizontalPadding // 1264
```

`WorkspaceView` 必须在**创建** `PageContainer` 时始终传入 `workspaceMaxContainerWidth`，不能等 `isWide` 后才传：`isWide` 依赖 GeometryReader，而 GeometryReader 能否达到阈值又依赖外层容器上限，二者不能循环决定。

```swift
PageContainer(maxContentWidth: LayoutConstants.workspaceMaxContainerWidth) {
    GeometryReader { geometry in
        let availableContentWidth = geometry.size.width
        let isWide = availableContentWidth >= LayoutConstants.workspaceWideContentThreshold
        // < 1164：单列；1164...1203：900/24/240；≥1204：920/24/260
    }
}
```

在窄窗口，父容器自然小于这个 max，GeometryReader 得到较小的实际宽度并走单列；不需要为窄模式再传回 880pt。这样才同时满足：唯一测量坐标、1164pt 临界点、≥1204pt 舒适尺寸和不压缩列宽。

## 同步文本修正

1. §2 的“≥1180pt 时提供辅助栏”已过时，应改为“当 `availableContentWidth >= workspaceWideContentThreshold` 时”，以免重新引入窗口总宽的错误解释。
2. §3.1、§6、§8、附录中“`maxContentWidth` 等于双列总宽”必须改为“等于双列内容预算 **加两侧 pageHorizontalPadding**”；Workspace 应始终传 `workspaceMaxContainerWidth`。
3. r4 文首的“已批准进入 P0”须等此修正落实后再保留。

## 批准门槛

完成以下三项后，**立即批准进入 P0，不再进行文档设计评审**：

- [ ] 文档和 `LayoutConstants` 区分 content width 与 PageContainer container width，并纳入两侧 padding。
- [ ] `WorkspaceView` 在构造 `PageContainer` 时无条件传入足够大的 `workspaceMaxContainerWidth`，由内部 GeometryReader 决定单/双列。
- [ ] 所有“1180pt”及“maxContentWidth 匹配双列总宽”的旧文本同步为真实内容宽度契约。
