## P0 前必须澄清的三项

### P0-1：1180pt 的测量坐标和尺寸预算

当前 §3.1 同时写“≥1180pt 主工作列 900–960pt + 240–280pt 辅助栏”，却没有规定 1180pt 是 **NSWindow 总宽度**、Sidebar 之后的内容宽度，还是 `PageContainer` 扣除水平 padding 后的可用宽度。

这三种宽度相差显著：主窗口已有约 180pt Sidebar，`PageContainer` 还会加入 60pt 水平 padding；若 1180 指总窗口宽度，内容区无法同时容纳最小的 `900 + 240 + gap`。若 `GeometryReader` 放在 `WorkspaceView` 内，拿到的又是 Sidebar 之后的内容宽度，语义与 `LayoutConstants.wideSecondaryThreshold` 的“窗口宽度”注释不一致。

**必须采纳的修订：**

1. 把阈值改名为 `wideWorkspaceContentThreshold`，并明确它量的是 `WorkspaceView` 在 padding 之后的可用内容宽度；或保留现名但明确它量的是窗口总宽度，重新按该预算设值。
2. 在文档和常量中给出单一公式：`mainColumn + gap + inspectorColumn <= availableContentWidth`。推荐起点为 `900 + 24 + 240 = 1164pt`，因此阈值不可低于 1164pt 的**实际内容宽度**。
3. 低于阈值必须退回单列，不能让辅助栏压缩主列、造成 920pt 下标题或按钮截断。

### P0-2：辅助栏不能“P0 布局、P1 填充”

§3.1 与 §6 的 P0 要求实现宽屏辅助栏，但 P1 才写“辅助栏内容填充”。这会产生一个空壳栏，既不能解决截图中的稀薄，也会让 P0 的视觉验收失真。

**必须采纳的修订：**将“Mac 名称、设备数/最近连接、Relay 自动连接摘要”作为辅助栏的最小内容一并移入 P0；P1 只允许做视觉细化，不得承担首次填充。数据源也须写明：设备数/最近连接来自既有 `DeviceStore`，Relay 摘要来自 `BridgeStatusViewModel`，Mac 名称应复用已保存的 `bridgeDisplayName`（或其现有配置源），不新增网络探测。

### P0-3：主 CTA 必须按运行状态定义

§2 说“常态下只允许一个 `.borderedProminent` 大按钮（首次使用为添加设备）”，但真实 `WorkspaceView` 已有 `.stopped/.idle` 的“启动工作站”和 `.crashed` 的“重新启动”。它们在没有设备、已有设备和异常时都可能是正确主动作，不能被“首次使用唯一”误读为降级。

**必须采纳的状态表：**

| 工作站状态 | 唯一主 CTA | 次级入口 |
| --- | --- | --- |
| Bridge 可连接、无设备 | 添加设备 | 无，或连接状态次级入口 |
| Bridge 可连接、已有设备、无异常 | 无 `.borderedProminent` 大按钮 | 查看设备、连接状态 |
| Bridge 已停止 / idle / sleeping | 启动工作站 | 查看设备、连接状态 |
| Bridge 崩溃 | 重新启动 | 帮助与诊断 |
| Bridge 运行但存在 backend / Relay 问题 | 不新增全局主 CTA；问题行内显示真实修复动作 | 查看设备、连接状态 |

这张表应进入设计文档 §2 或 §7，并以现有 `BridgeRuntimeStatus` 分支为真值，不另造 UI 状态。

## 需在 P1/P2 落地时遵守的澄清

### 连接入口的“唯一性”应指唯一目的地，不是唯一按钮位置

修订版同时允许工作站 Relay 行显示“连接状态”次级按钮，且 Toolbar 也有该按钮。这是正确的，只要二者打开**同一个** Sheet。建议把“唯一连接相关动作入口”改为“唯一连接相关**目的地**”，避免实现者为了字面唯一而删除有上下文价值的次级按钮，或反过来新增第二个 Sheet。

### 默认 endpoint 只对官方默认值隐藏

P1 要求删除默认不必要的 endpoint 是对的，但必须保留已选择“自定义中继”时的 endpoint、保存/恢复官方默认和真实错误。否则会把高级用户的实际配置变成不可核对的黑箱，也会违反“真实失败直接暴露”的工程约束。

### 日志模式必须区分显示集与复制集

P2 应明确维护 `visibleLogs`（默认 30 / 展开 200）和 `rawLogs`（最近 200 行）的关系：展开和“复制原始日志”都使用完整的 `rawLogs`；30 行仅限制屏幕渲染，不能影响读取、复制或支持信息。当前 `DiagnosticsSheet` 只有 `showRawLogs` 布尔值和完整 logs 列表，实施时需要这一明确拆分，避免“默认 30 行”被误实现成只加载 30 行。

## 不采纳项复核

修订版明确拒绝的四项均应继续维持：

- **全区域卡片化 / 多张状态卡**：会把工作站变回 dashboard，且无法解决重复信息。
- **查看设备与连接状态双主 CTA**：会稀释用户唯一明确的下一步。
- **复制配对链接升为主操作**：会与 QR 的标准配对路径争夺层级。
- **TextEditor 或装饰性格式替换原始日志**：会降低原始证据的可复制性和可信度。

本轮没有新增需要拒绝的产品方向；上述三项 P0 澄清属于补足规格，不是推翻修订版。

## 验收补充

在修订版 §8 的清单之外，P0 自验需增加：

- 使用定义后的“实际内容宽度”临界值前后各验证一次：阈值下单列、阈值上两列，均无压缩/横向截断。
- 宽屏辅助栏在 P0 即包含三类非重复事实，不存在空壳。
- 逐一验证上文五种工作站状态时，至多一个 `.borderedProminent` 主 CTA 且动作与 runtime 状态一致。
- Toolbar 和工作站内的“连接状态”均进入同一 `RemoteAccessView` Sheet；未新增 presentation state 或第二目的地。

完成 P0-1、P0-2、P0-3 的文字与常量契约修订后，**批准进入 P0 实现**。此时方案已不再只是视觉偏好，而是能在现有 SwiftUI 状态、窗口预算和运行时操作约束下稳定交付的实施规格。
