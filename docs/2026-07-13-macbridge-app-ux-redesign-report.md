# CordCode Link macOS App UX 重设计报告

日期：2026-07-13
范围：`MacBridge/` 桌面 App 的信息架构、页面布局、交互优先级与视觉方向。本文是下一轮实现的产品设计基线，不改变 Bridge、配对或 Relay 协议。

## 结论

CordCode Link 应被用户一眼理解为：**让 iPhone / iPad 能随时接入这台 Mac 上 AI 编程工具的安全工作站**。当前版本已经具备所需功能，但呈现方式更像运行时控制台：把「总览、设备、远程访问、设置、诊断」并列成五个技术页，用户必须先理解 Bridge、Relay、Tailscale、端口和 backend，才知道下一步该做什么。

重设计不应在现有页面上继续加卡片或说明文字，而应改为一个围绕用户任务组织的工作台：

1. 第一次打开：完成“检查此 Mac → 添加设备”这一条连续路径；
2. 日常打开：一眼确认“现在能不能从手机继续工作”；
3. 出现问题：在出错位置给出可执行修复动作；
4. 少数高级需求：不干扰主流程，但可直接进入，绝不藏进多层菜单。

## 现状审视

本报告基于 2026-07-13 的五张实际界面截图及当前 SwiftUI 实现（`ContentView`、`BridgeStatusView`、`PairingView`、`RemoteAccessView`、`SettingsView`）完成。

### 已经成立的基础

- 深色主题、系统字体、低饱和蓝色选中态与 SF Symbols 已建立了克制的 macOS 基调。
- 配对流程拥有 QR、手动码、倒计时、审批和失败态，功能链路完整。
- 首页已经能汇总运行、AI 工具、设备与 Relay 状态；配对按钮也已放在高位。
- Relay 作为默认远程路径、Tailscale / VPS 作为附加路径的产品策略是正确的。

### 阻碍成熟感的关键问题

| 观察 | 对用户的影响 | 应采取的方向 |
| --- | --- | --- |
| 首屏标题是“总览”，内容是四组状态行，右侧大面积留白。 | 用户看见的是管理后台，不知道产品能为自己做什么。 | 首屏改为“这台 Mac 已准备好供你的设备继续工作”的任务面，而不是状态表。 |
| 五个一级导航按实现模块划分。 | “远程访问”和“设置”的边界不清；“诊断”过早进入主导航。 | 一级导航只保留日常任务；低频配置与诊断转为原生 Settings / Help & Diagnostics。 |
| “远程访问”内部再次出现左栏 + 右栏、自动/其他连接、多个详情页。 | 单一配置页形成二级导航；用户必须在 LAN、Relay、Tailscale、VPS 之间做本不需要做的选择。 | 默认只解释并展示实际采用的自动连接；高级方式在同页按需展开。 |
| 设备页将按需配对与已授权设备置于同一页面；当前 QR 已只在用户点击“配对新设备”后出现。 | 现有状态机已避免 QR 常驻，但首次配对、等待确认与日常设备管理仍共享同一画布，任务起止不够明确。 | 保留按需展示 QR 的正确做法；将既有状态机承载为聚焦的配对工作区 / Sheet，并把设备清单留作日常次级内容。 |
| 设置页暴露了自动重启、OpenCode source、认证命令、密码等运行时细节。 | 一般用户会误以为必须配置；敏感/高级动作的可见性过高。 | 采用渐进披露：默认托管无需配置，只有主动选择“使用自己的 OpenCode 服务”才展开表单。 |
| 原始日志目前由 `ContentView` 内联的 `logsTab` 呈现在一级“诊断”。 | 产品把维护者需求与普通用户日常操作等权，且诊断 UI 尚未形成独立产品区域。 | 从 `ContentView` 抽离为帮助入口中的独立工作表；先给健康摘要、再给复制诊断包与原始日志。 |

此外，当前全局 `PageContainer` 将内容固定在约 820pt 宽的居中窄列；`RemoteAccessView` 的现有左/右二级导航也受这一硬约束限制。在大屏上它放大了空白，也削弱了“这台 Mac”的在场感。空白本身不是问题，但需要被明确的主动作、设备状态和空间层级组织，而不是让表单漂浮在画布中央。`PageContainer` 的宽度策略因此是 P0 必须先处置的工程前提，而不是 P1 可以局部绕开的样式细节。

## 设计原则

### 视觉与交互主张

- **视觉主张**：深色、安静、可靠的本地工作站；不是服务器控制台，也不是充满渐变卡片的 SaaS 仪表盘。
- **内容主张**：第一屏只回答三个问题——是否已就绪、哪些设备可用、下一步做什么。
- **交互主张**：常用动作永远在当前面板直接可达；复杂配置使用同页展开、Sheet 或独立原生设置窗，不把用户推入二、三级导航。
- **视觉节奏**：一处蓝色只用于主动作；绿/黄/红仅表达运行健康；用版式、细分隔线和留白代替大面积圆角卡片。

### 信息层级规则

1. 首次使用者不应看到“Relay endpoint”“端口”“OpenCode password”。
2. 系统正常时不展示冗余细节；异常时才展开原因、影响范围与修复按钮。
3. 每个页面只能有一个主要动作，且动作使用动词：`添加 iPhone 或 iPad`、`重新检查`、`恢复连接`。
4. 不让用户选择产品已经可以自动选择的路径。Relay 是默认安全远程路径；LAN 是自动加速路径；Tailscale 和自定义地址归入高级选项。
5. 破坏性操作（停止 Bridge、撤销设备、关闭 Relay）保留确认，但确认框要说明结果和恢复路径。

## 新的信息架构

主窗口收敛为两个日常入口；不再把技术后台直接暴露为平级目的地。

| 位置 | 新名称 | 用户任务 | 取代 / 吸收 |
| --- | --- | --- |
| Sidebar | **工作站** | 看就绪状态、添加设备、处理阻塞项。 | 现“总览”，并吸收 AI 工具健康概览与默认远程状态。 |
| Sidebar | **设备** | 添加、确认、管理已授权 iPhone/iPad/Web。 | 现“设备”。 |
| Toolbar | **连接状态** | 查看当前自动连接是否可用；必要时修改高级连接。 | 现“远程访问”。 |
| App 菜单 / `⌘,` | **设置…** | 外观、名称、自动恢复、OpenCode 高级接入。 | 现“设置”。 |
| Toolbar Help 菜单 / `⌘⇧D` | **帮助与诊断…** | 查看健康摘要、导出支持信息、查看日志。 | 现“诊断”。 |

这里的“移出 Sidebar”不是隐藏功能：连接、设置、诊断各自仍是一跳可达的原生窗口或 Sheet。这样主导航表达用户每天要做的事，而不是代码模块的目录。

## 核心界面方案

### 1. 工作站（默认页）

首屏应采用稳定的三段纵向工作面，而非 dashboard 卡片拼贴。窗口建议最小宽度提升至 920pt，内容左对齐于 760–880pt 宽的工作列；右侧只在大于约 1180pt 时出现只读的连接健康辅助信息，不新增独立目的地或操作。

```text
CordCode Link                                      [添加设备]

这台 Mac 已准备好
从 iPhone、iPad 或 Web 继续使用 Mac 上的 AI 编程工具。

● 可以连接                         1 台设备 · 上次连接 14 分钟前
  Bridge 正在运行 · Claude Code、Codex、Grok Build、OpenCode 已就绪
  [查看设备]                                             [连接状态]

需要留意（只在异常时出现）
! Codex 需要重新连接                               [重新检查]

可用的 AI 工具
Claude Code     就绪
Codex           就绪
Grok Build      就绪
OpenCode        就绪

安全连接
Relay 已开启，离开局域网后仍可安全使用。
```

- `可以连接 / 还差一步 / 需要处理` 是核心健康结论；用文字和状态点共同表达，不使用只靠颜色的成功/失败。
- 正常状态不显示端口、版本、endpoint；这些进入“连接状态”或“帮助与诊断”。
- “停止 Bridge”从首屏移入状态行的省略号菜单，或仅在异常/维护场景出现；不能与“添加设备”竞争。
- `连接状态`是工作站中**唯一的连接入口**，与 Toolbar 的“连接状态”打开同一个 Sheet；“安全连接”区块仅呈现当前 Relay 结果，不再放第二个详情按钮。异常状态可在原因旁直接给出“连接状态”动作，但仍指向这个同一 Sheet。
- 工作站的行内修复动作（如“需要留意”段的 `[重新检查]`）属于与“连接状态”同模式的行内 CTA：复用 `BridgeStatusViewModel` 既有健康探测能力，不新造独立诊断动作，落在对应异常原因旁边。
- AI 工具一行只显示名称与健康摘要。正常时没有“管理工具”总入口，也不暗示每个工具都需要配置；`测试连接`是次级 hover/行内动作，失败时替换为明确的修复 CTA。低频的 OpenCode 接入配置统一由 `设置… → 高级` 承担。
- 若没有任何设备，首屏的状态段直接替换成首次使用引导：`连接你的第一台设备` + 一句说明 + `添加 iPhone 或 iPad` 主按钮；不要先要求用户研究远程访问。

### 2. 设备：把配对当成一个可完成的任务

设备页顶部常驻 `添加设备` 主按钮。当前 `PairingView` 已通过 `uiState` 做到“仅按钮 → 点击后显示 QR → 等待/审批/结果”的按需流；重设计应保留这条真实状态机，改为在当前页的聚焦工作区或 560–640pt 宽 Sheet 中承载它，而不是误把目标定义为“消除常驻 QR”。

配对工作区的建议结构：

```text
添加设备                                                  [关闭]
1 扫描二维码  →  2 在 Mac 上确认  →  3 开始使用

      [ QR code ]             用 iPhone 或 iPad 的 CordCode 扫描
                               手动代码 266144   [复制]
                               有效期 04:52

                               等待设备连接…
                               [复制配对链接]  [连接方式详情]
```

- 上方显示步骤轨迹，但只高亮当前步骤；这能让二维码不再像一个没有解释的技术对象。
- iOS / Web 的 QR 选择改为清晰的文本切换 `iPhone / iPad`、`Web`，仅当 Web 配对真实可用时显示。
- 设备 claim 到达时，工作区立即变成“`Alice 的 iPhone 请求访问`”及 `允许`（蓝色主按钮）/`拒绝`，不能要求用户先理解“claimed”。系统通知仍可保留作为窗口不在前台时的补充。
- 已授权设备以紧凑列表置于下方；每一行展示名称、平台、最近连接。当前实现已经在撤销后弹出二次确认；重设计只把列表上的“撤销授权”改为行尾 `…` 菜单，并在既有确认中补齐“将断开此设备、之后需要重新配对”的结果与恢复文案，而不改变撤销的安全行为。
- 工作站首次使用引导的“添加 iPhone 或 iPad”与设备页顶部的“添加设备”复用同一个配对工作区组件与触发路径，不实现成两套独立配对 UI。
- 配对工作区的承载方式（当前页聚焦工作区或模态 Sheet）由实现按 SwiftUI 成本选择；选 Sheet 时配对期间为模态，选同页工作区时需保证设备列表上下文仍可见。状态机、审批、二维码与通知行为在两种承载下保持一致。

### 3. 连接状态：自动优先，高级按需

它不再是一个含二级导航的“远程访问”页，而是从 Toolbar 或工作站健康段的同名动作打开的**同一个**连接检查 Sheet（建议 680–760pt）。不存在独立的“连接详情”目的地；默认展现真实路径和安全解释：

```text
连接状态                                              [刷新]

安全远程连接          ● 已开启
Relay 会在离开局域网时自动保持连接；内容始终端到端加密。
当前状态：已配置                                        [关闭…]

本地网络              ● 可用
同一 Wi‑Fi 下会自动使用更快的直接连接。

高级连接                                          [显示高级选项]
```

展开“高级连接”后，同一页面顺次出现 Tailscale 和自定义地址，而不是先让用户在左栏选择一种“连接方法”。每项都采用“状态 → 一句适用条件 → 开关/输入 → 保存”的顺序。默认 Relay endpoint 不需要展示；选择“自定义中继”后才显示地址输入和“恢复官方默认”。

这符合实际产品语义：LAN 与 Relay 都是自动路径，不是互斥模式；Tailscale/VPS 是特殊需求，不能让它们看上去与默认 Relay 同等推荐。

### 4. 设置：原生、低频、渐进披露

将现 Settings 从 Sidebar 移入 macOS 标准 `Settings` 窗口，左侧只使用两项：`通用`、`高级`，没有再嵌套的导航层级。

- **通用**：语言、外观、此 Mac 的名称、自动恢复。每项用系统 Form 样式，保存采用即时提交或字段级确认，避免整页“保存”。
- **高级**：OpenCode 接入。默认说明为“CordCode Link 正在自动管理 OpenCode，无需操作”。只有选择“使用自己的 OpenCode 服务”后显示 URL、验证和认证信息；legacy 64667 不作为普通可见选项，应放在明确的兼容说明之后。
- 密码、启动命令、重新生成等敏感/维护操作只在相关模式下显示；绝不在默认状态占据视觉高度。

### 5. 帮助与诊断：先结论，后日志

`帮助与诊断`打开的窗口先提供一个可读的健康摘要：Bridge、连接、AI 工具三个结果，以及针对失败项的“复制支持信息”按钮。原始日志是同窗的第二段或通过 `查看原始日志` 打开，保留现有复制功能和等宽字体，但不再作为日常导航入口。

当某项失败，错误文案统一采用：**发生了什么 → 对你有什么影响 → 现在可以做什么**。例如：`Codex 暂时不可用。你仍可从手机使用 Claude Code 和 OpenCode。重新检查，或在 Mac 上启动 Codex。`

## 关键状态与文案

| 产品状态 | 标题 | 说明 | 主动作 |
| --- | --- | --- | --- |
| 首次使用 | 连接你的第一台设备 | 扫描一次，即可从 iPhone 或 iPad 继续使用这台 Mac 的 AI 工具。 | 添加设备 |
| 全部就绪 | 这台 Mac 已准备好 | 你的设备可通过局域网或安全 Relay 连接。 | 添加设备 / 查看设备 |
| 无可用 AI 工具 | 还差一步才能开始 | CordCode Link 正在运行，但还没有可用的 AI 工具。 | 重新检查 |
| Relay 暂不可用 | 远程连接暂不可用 | 同一 Wi‑Fi 下仍可连接；离开局域网后暂不能继续使用。 | 查看连接状态 |
| Bridge 停止 | 工作站已暂停 | 已授权设备暂时无法连接此 Mac。 | 启动工作站 |

避免使用“总览”“配置状态”“自动连接（默认使用）”等内部管理语言作为首要文案；它们可用于技术详情标题，但不能承担产品解释。

## 视觉系统与行为细节

- 继续采用系统 San Francisco 字体和 SF Symbols；标题使用 `.title` / `.title2` 的语义字体，不固定像素字号。
- 统一 8pt 间距节奏；主要区块间 28–32pt，行内 8–12pt。用 `Divider` 与背景层级分段，不新增无意义的浮卡阴影。
- 内容宽度不再固定为单一 820pt：主工作列应随窗口扩到 880pt；宽屏才增加次级信息栏。窄窗口则保持单列、标题和主要操作同一行可见。
- 主按钮仅有一个（蓝色）；`bordered` 用于次要操作，`borderless` 用于复制、详情、行内操作。破坏性按钮只在确认上下文中强调。
- 状态转换采用 160–220ms 淡入/高度动画：配对从“等待”切到“请求确认”、健康问题出现/解决、展开高级连接。不要为纯装饰添加持续动画；`Reduce Motion` 时取消位移，只保留状态替换。
- 所有状态颜色必须配合图标与文字；QR、倒计时、复制成功、进度与错误应具有 VoiceOver 标签和键盘焦点顺序。

## 实施分期与验收

### P0：结构与首屏（最高价值）

1. 将 `ContentView` 的 Sidebar 收敛为“工作站 / 设备”，把 Settings 接入原生 Settings scene，把连接状态和诊断改为 Toolbar/Sheet 入口。
2. 用新的 `WorkspaceView` 替换当前 `BridgeStatusView` 的四段状态表，先完成首次使用、全就绪、异常三种状态。
3. 先将 `PageContainer` 改为可配置宽度（或让连接 Sheet 不使用它），并把窗口最小宽度与工作列宽度作为 P0 的布局契约；不能等到 P1 单独修改 `RemoteAccessView` 才处理。
4. 明确状态归属：`ContentView` 当前拥有设备列表、设备加载/错误状态以及日志状态。随功能抽离将设备状态迁入工作站/设备的共享 store 或专属 ViewModel；日志状态随诊断工作表抽离迁入其专属 ViewModel。两者继续使用同一 Management API 数据源，不把它们错误塞入 `BridgeStatusViewModel`。
5. 保持现有 `RuntimeManager`、Management API、`BridgeStatusViewModel` 的运行状态职责与配对协议不变；本阶段重组 View、状态所有权和用户文案，不改变运行语义。

验收：第一次启动后，无需理解 Bridge / Relay / Tailscale，用户可在一屏内知道产品用途并开始添加设备；已配对用户能在一屏内确认可用性。

### P1：设备与连接收口

1. 将 `PairingView` 的既有按需状态机映射为任务式配对工作区 / Sheet，保留审批、二维码、手动码和通知行为。
2. 将 `RemoteAccessView` 改为单页面“连接状态”；Relay 和 LAN 默认可见，Tailscale 与自定义地址迁入同页的高级展开区。
3. 在 P0 宽度策略落地后，移除页面内的可拖拽次级侧栏和窄窗口“返回详情”模式，并归并现有 `selectedMethod` 状态而不改变连接配置语义。

验收：用户不需要在“自动连接 / 其他连接”的二级列表中选择；每个高级动作从连接状态窗口一跳可达。

### P2：设置、诊断与细节抛光

1. 设置按“通用 / 高级”重组，OpenCode 默认管理态只呈现确认信息。
2. 从 `ContentView.logsTab` 抽离 `DiagnosticsSheet`；诊断改为摘要优先，增加可复制的脱敏支持信息；原始日志保留为按需查看。
3. 补齐空状态、错误状态、VoiceOver、键盘快捷键与动态类型检查。

验收：正常用户看不到密码、命令、endpoint 和原始日志；遇到故障时能从同一位置获取明确下一步。

## 工程边界与建议

- 本次是 UI/UX 重构，不能借机改变 Relay 默认、配对安全、backend 运行模型或诊断的真实语义。
- 组件建议按“产品区域”拆分（`WorkspaceView`、`DeviceWorkspaceView`、`ConnectionStatusSheet`、`DiagnosticsSheet`），不要继续把全部状态拼接在 `ContentView`。
- 本报告刻意不推荐 mock、假状态或用于演示的生产 fallback；Preview 可以建立隔离 fixture，真实 UI 仍必须消费现有 ViewModel 状态。
- 实现时先做 Swift 单元测试和 macOS 定向 build。视觉自动化、snapshot、simulator automation 或真机 UI 验收仍须在执行前取得 owner 的明确授权。

## 成功标准

重设计完成后，一位第一次安装的用户应能在 10 秒内回答：

> 这是把我的 Mac 上 AI 编程工具安全带到 iPhone / iPad 的应用；现在它是否已经准备好；下一步按哪里连接设备。

一位已使用的用户应能在 3 秒内确认：这台 Mac 是否可连接、是否有需要处理的问题，以及自己的设备是否仍被授权。若不能做到这两点，界面再精致也仍是工程控制台，而不是成熟产品。
